
# 手动推送页面紧凑版 UI 设计与前端改造方案

> 页面：手动推送  
> 目标：只改前端展示层，不修改后端接口和业务逻辑；保留数据源选择、推送模式、目标日期、科室过滤、审计类型、类型并行、预览模式、异步执行、批量参数、Dify 节点配置、SQL 查询结果预览等现有能力。  
> 设计方向：紧凑型推送任务配置台，突出“先配置 → 预览 → 查询 SQL → 开始推送 → 查看进度”的操作路径。

---

## 1. 结合代码与截图的现状判断

GitHub 当前基线代码里，`activeMenu === 'push'` 的原始“数据推送”页面主要是一个简单表单：查询日期、指定科室、Dry Run 开关、执行推送按钮，以及执行结果展示。也就是说，原有核心逻辑是“选择日期/科室 → 执行推送 → 展示结果”。截图中的本地页面已经扩展出更多能力，包括 Oracle/PostgreSQL 数据源、单日/时间范围、审计类型、多类型并行、预览模式、异步执行、批量设置、Dify 节点配置和 SQL 结果预览。因此本次设计应以截图为准，但保留旧代码兼容：不要破坏 `submitPush`、`pushForm`、`pushResult` 等原有功能，只在前端布局和字段组织上优化。

---

## 2. 当前界面主要问题

1. 页面留白太多，表单区域垂直距离偏大。
2. “数据源、推送模式、目标日期、科室过滤、审计类型”等配置项分散，用户需要上下扫。
3. “类型并行、预览模式、异步执行”是重要开关，但视觉上和普通输入项混在一起。
4. “批量推送设置”标题线比较弱，参数没有形成清晰卡片组。
5. “开始推送 / 预检 / 查询 SQL 结果 / 查看最近进度”按钮放在中间，缺少固定操作区。
6. Dify 节点配置区域高度较大，但未配置时可以压缩成空状态卡。
7. SQL 查询结果预览区可继续保留，但应做成“预览结果工作台”，表格区域尽量多显示记录。
8. 页面整体更像表单堆叠，不像“任务配置与执行台”。

## 3. 推荐页面定位

```text
手动推送任务配置台
```

核心目标：

```text
1. 一屏完成核心参数配置
2. 高级参数默认折叠或紧凑展示
3. 操作按钮始终清晰
4. 预览 SQL 结果尽量多显示患者记录
5. Dify 节点配置不抢主流程空间
6. 开始推送前让用户明确：数据源、范围、审计类型、预览/正式、异步/同步
```

## 6. 具体设计建议

### 6.1 基础配置卡

建议将截图中的基础字段合并成 2～3 行网格：

```text
第 1 行：
数据源 | 推送模式 | 目标日期/日期范围

第 2 行：
科室过滤 | 审计类型

第 3 行：
类型并行 | 预览模式 | 异步执行
```

这样可以把当前较长的表单压缩到 180px 左右高度。

### 6.2 批量推送设置

截图中“批量推送设置”可以改为四个小输入卡：

```text
并发工作线程
空结果重试次数
重试退避毫秒
目标分配策略
```

每个卡片高度控制 44px～52px，不要用大表单行。

### 6.3 固定操作条

将按钮集中到一条操作栏：

```text
开始推送 | 预检 | 查询 SQL 结果 | 查看最近进度
```

按钮建议靠左，右侧显示当前任务摘要：

```text
Oracle · 单日推送 · 2026-06-14 · 3 个审计类型 · 异步执行
```

### 6.4 Dify 节点配置

未配置节点时不要占太高空间。建议空状态：

```text
当前未配置 Dify 节点，将使用系统配置中的单个 Dify。
```

右上角保留：

```text
载入默认 | 新增节点 | 载入已保存 | 保存配置
```

### 6.5 SQL 查询结果预览

这个区域应是列表优先：

```text
只看未推送
患者ID/姓名/住院号搜索
推送勾选记录
表格
```

表格建议列：

```text
选择
推送标记
最近状态
患者ID
姓名
住院号
次数
在院科室
合并明细
病历时间
诊断
操作
```

如果没有数据，空状态保持紧凑，不要占过多高度。


# 审计类型管理页面 UI 重构方案

> 页面：审计类型管理  
> 目标：只改前端展示层和交互组织，不修改后端接口与现有业务逻辑。  
> 设计定位：从“审计类型表格配置页”升级为“规则编排控制台”。

---

## 1. 当前页面问题

当前页面功能已经比较完整，但信息层级较混乱，主要问题如下：

1. 审计类型定义、数据源 SQL、Payload、Dify 工作流、安全标志、测试按钮全部平铺在表格里。
2. 每行按钮过多：编辑、克隆、测试数据源、测试 Dify、删除，视觉噪音较大。
3. Dify 的 Key、URL、入参、来源等调试信息直接暴露在主列表，不利于快速理解规则状态。
4. “运行解析摘要”区域信息很多，但与上方规则列表关系不够清楚。
5. 当前页面更像 Excel 配置表，不像可持续维护的规则编排平台。
6. 表格纵向和横向都偏拥挤，后续审计类型增加后维护成本会越来越高。

---

## 2. 推荐页面定位

建议将该页面定位为：

```text
AI 质控规则编排控制台
```

页面核心不再是“显示一张大表”，而是：

```text
规则列表 → 规则详情 → SQL/Payload/Dify 配置 → 测试与运行状态
```

---

## 3. 推荐信息架构

### 3.1 页面整体结构

```text
顶部标题操作条
├─ 审计类型管理
├─ 副标题：规则定义、SQL 数据源、Payload 构建、Dify 工作流和测试调试
└─ 新建类型 / 导入配置 / 刷新

状态概览条
├─ 审计类型总数
├─ 已启用
├─ 默认调度
├─ 配置告警
├─ SQL 已配置
└─ Dify 已配置

主体三栏
├─ 左侧：审计类型列表
├─ 中间：当前规则配置详情
└─ 右侧：执行链路与风险摘要
```

---

## 4. 主体三栏设计

### 4.1 左侧：审计类型列表

左侧只展示规则核心状态，不展示 SQL 与 Dify 细节。

建议字段：

```text
编码
名称
启用状态
默认调度
配置风险
最近测试状态
```

推荐展示示例：

```text
admission_vs_first_progress
入院与首次病程核查
启用 / 默认调度 / 告警 1

surgery_chain
围手术期核查
启用 / 默认调度 / 正常
```

### 4.2 中间：规则详情

中间区域使用 Tab：

```text
基础信息
数据源 SQL
Payload 构建
Dify 工作流
测试调试
```

每个 Tab 的职责：

| Tab | 内容 |
|---|---|
| 基础信息 | 编码、名称、描述、启用状态、默认调度、必填标签 |
| 数据源 SQL | 数据源、SQL 模板、输出字段、变量说明、测试 SQL |
| Payload 构建 | JSON 入参模板、字段映射、默认值、缺失值处理 |
| Dify 工作流 | workflow key、api key、url、超时、重试、输出解析 |
| 测试调试 | 测试数据源、测试 Dify、测试结果、耗时、错误日志 |

### 4.3 右侧：执行链路摘要

右侧展示运行态和风险态：

```text
执行链路
SQL → Payload → Dify → 解析 → 审计展示

配置完整性
SQL：已配置
展示：有
密钥：有
URL：已配置

最近测试
数据源：通过
Dify：通过
解析：通过

风险提示
缺少默认 SQL
Dify 未测试
Payload 字段缺失
```

## 5. 操作按钮收敛

当前每行操作较多，建议主列表只保留：

```text
编辑
更多
```

更多菜单中放：

```text
克隆
测试数据源
测试 Dify
导出 JSON
删除
```

在详情页右上角保留：

```text
保存
测试 SQL
测试 Dify
克隆
删除
```

这样既保留功能，又降低主表格视觉压力。

## 8. 前端改造建议

### 8.1 不改后端接口

本次只调整前端：

```text
1. 不修改现有保存接口
2. 不修改测试数据源接口
3. 不修改测试 Dify 接口
4. 不修改删除接口
5. 不修改审计类型 JSON 结构
```

### 8.2 建议新增前端状态

```javascript
const selectedAuditType = ref(null)
const activeAuditTab = ref('basic')
const showRuntimePanel = ref(true)

```

### 8.3 建议组件拆分

```text
AuditTypePage
├─ AuditTypeToolbar
├─ AuditTypeSummaryBar
├─ AuditTypeList
├─ AuditTypeDetailTabs
```

## 9. 本地 AI 执行提示词

```text
请对“审计类型管理”页面进行前端 UI 重构，只改前端，不修改后端接口和业务逻辑。

目标：把当前表格平铺式页面改成“AI 质控规则编排控制台”。

要求：
1. 保留现有审计类型列表、新建类型、编辑、克隆、测试数据源、测试 Dify、删除功能。
2. 不修改后端接口和已有 JSON 配置结构。
3. 页面顶部使用 46px 标题操作条，右侧放“新建类型 / 导入配置 / 刷新”。
4. 增加一行状态概览条，显示：审计类型总数、已启用、默认调度、配置告警、SQL 已配置、Dify 已配置。
5. 主体改为三栏布局：
   - 左侧：审计类型列表，只显示编码、名称、启用、默认调度、风险状态。
   - 中间：当前规则详情，使用 Tab：基础信息、数据源 SQL、Payload 构建、Dify 工作流、测试调试。
   - 右侧：执行链路摘要，显示 SQL → Payload → Dify → 解析 → 审计展示，以及配置完整性和最近测试结果。
6. 主列表不要直接展示 Dify Key、URL、SQL 细节，这些放入详情 Tab。
7. 每行操作按钮收敛为“编辑 / 更多”，更多中放克隆、测试数据源、测试 Dify、删除。
8. SQL 编辑器高度控制在 220px～260px，增加格式化、复制、测试 SQL 按钮。
9. Payload 构建区域显示 JSON 模板和字段映射说明。
10. Dify 工作流区域隐藏 API Key，只显示已配置状态，支持测试 Dify。
11. 测试调试区域展示测试输入、返回 JSON、耗时、错误日志。
12. 保证保存、测试、删除、克隆等原功能不受影响。
```

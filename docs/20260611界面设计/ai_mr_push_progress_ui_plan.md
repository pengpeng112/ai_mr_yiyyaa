# 推送进度页面紧凑版 UI 设计与前端改造方案

> 页面：推送进度  
> 目标：将当前分散在手动推送页和调度历史中的任务进度信息集中展示，便于用户查看批量推送任务的实时/历史进度。  
> 设计方向：紧凑型任务进度工作台，突出“正在运行、最近完成、失败重试、任务详情”。

---

## 1. 设计定位

推送进度页面定位为：

```text
推送任务进度跟踪台
```

核心目标：

```text
1. 一眼看到当前是否有推送任务在运行
2. 快速查看最近完成的任务及其成功/失败数量
3. 支持按状态、日期、触发方式、审计类型筛选
4. 点击任务查看详细进度和诊断信息
5. 对运行中任务支持停止（如果后端接口支持）
```

> **已确认（2026-06-14）**：推送进度作为新增独立页面，放入左侧菜单“推送管理”组，位于“定时任务”之后。

---

## 2. 基于现有代码的改造边界

当前已有后端能力（参考 `push.js` / `scheduler.js`）：

```text
- 手动推送提交后返回 task_id
- /api/push/task/{task_id}/progress 查询任务进度
- /api/push/task/latest 查询最近任务
- 调度器执行历史 /api/scheduler/history
- 手动推送结果包含 total/success/failed/skipped/diagnostics
```

因此本次只改前端：

1. 不新增后端推送执行逻辑。
2. 不修改已有手动推送接口。
3. 复用现有任务进度查询接口。
4. 可复用 `schedulerHistory` 或新增 `pushTaskList` 作为数据源。

## 7. 任务详情抽屉

点击任务行后，右侧弹出抽屉展示：

```text
1. 任务摘要：任务ID、触发方式、状态、开始/结束时间、总耗时
2. 进度条：已处理 / 总数，成功 / 失败 / 跳过
3. 审计类型分布：每个审计类型的处理数量
4. 诊断信息：warnings/diagnostics
5. 跳过原因统计（如果有）
6. 查看关联推送日志：跳转推送日志并带上 task_id 或时间窗过滤
```

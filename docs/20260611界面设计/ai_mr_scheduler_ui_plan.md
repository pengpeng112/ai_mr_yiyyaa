
# 定时任务页面紧凑版 UI 设计与前端改造方案

> 页面：定时任务  
> 目标：只改前端展示层，不修改后端接口和业务逻辑；保留状态刷新、立即触发、启用/停用调度、配置保存、执行历史、配置风险等功能。  
> 设计方向：紧凑型调度任务工作台，突出“运行状态、下一次执行、最近执行结果、配置风险、任务历史”。

---

## 1. 结合代码的功能边界

后端已有 `/api/scheduler` 路由，支持：

```text
GET  /api/scheduler/status    调度器状态
POST /api/scheduler/start     启用定时任务
POST /api/scheduler/stop      停用定时任务
POST /api/scheduler/trigger   立即触发一次推送
GET  /api/scheduler/history   执行历史
```

后端调度器使用 APScheduler，任务 ID 为 `daily_push`，启动时读取配置中的 `scheduler.enabled` 与 `scheduler.cron`；启用时调用 `update_scheduler(True, cron)`，停用时移除定时任务；立即触发会后台启动线程执行任务。

因此本次前端只需要围绕这些已有能力重排页面，不建议改后端接口。

---

## 2. 当前界面主要问题

从截图看，当前页面功能比较完整，但还可以进一步优化：

1. 顶部“定时任务工作台”区域偏高，占用了较多首屏空间。
2. 4 个状态卡片较大，但核心信息并不多。
3. “最近一次执行”可以继续保留，但应做成更紧凑的状态条。
4. Tab 区域可保留，但内容卡片内部表单间距偏大。
5. 出院终末说明提示条占用高度较大，建议压缩成可折叠说明。
6. 每日执行时间、审计类型、科室配置等表单项可以组成网格，减少纵向空间。
7. 底部操作按钮建议固定在配置卡右下或形成操作条。
8. 执行历史与配置风险应更容易切换和查看。

## 6. 具体设计建议

### 6.1 顶部标题操作条

不要使用大 Banner，改为紧凑标题条：

```text
定时任务
病历质控自动调度 · 每日增量 · 出院终末 · 手动触发
```

右侧按钮：

```text
刷新状态
立即触发
查看历史
```

### 6.2 状态细条

将当前 4 个大卡改为一行状态条：

```text
调度器 运行中 | 每日增量 已启用 08:32 | 出院终末 已启用 13:22 | 配置风险 2项 | 下一次执行 2026-06-15 08:32
```

颜色：

```text
运行中/已启用：绿色
停用：灰色
风险：橙色
失败：红色
```

### 6.3 最近执行条

当前“最近一次执行”卡片可以压缩为：

```text
最近执行：2026-06-14 13:50:49 | 查询日期 2026-06-13 | 总记录 132 | 成功 130 | 失败 0 | 耗时 1149s | completed
```

### 6.4 Tab 区

保留当前：

```text
每日增量
出院终末
立即触发
执行历史
配置风险
```

但高度控制到 36px，激活态使用蓝色下划线或浅蓝背景。

### 6.5 出院终末配置区

当前说明提示较长，建议：

```text
出院终末质控说明
出院终末质控查询昨天出院患者，仍使用原有审计类型；病程与护理按同一天匹配，不做全周期笛卡尔匹配。
```

压缩成 44px 高提示条，增加“展开说明”。

### 6.6 表单网格

建议：

```text
第 1 行：启用调度 | 每日执行时间
第 2 行：审计类型 | 质控科室
第 3 行：科室编码/名称补充输入
```

按钮固定在右下：

```text
保存配置
启用出院终末调度
停用出院终末调度
```

### 6.7 执行历史

执行历史建议使用紧凑表格：

```text
执行时间
触发类型
查询日期
总数
成功
失败
耗时
状态
操作
```

表格行高 38px～42px。

### 6.8 配置风险

配置风险不要只显示“2项”，应展示：

```text
风险级别
风险项
说明
建议操作
```

例如：

```text
信息：未配置 Dify 多节点，仅使用默认节点
警告：失败数连续升高，建议检查 Dify / Oracle 连接
```

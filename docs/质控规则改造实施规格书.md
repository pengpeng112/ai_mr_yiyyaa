**不需要改造的（已确认正确）**：
- 规则 1-2 在院质控（同日交叉）

- 既有跳过策略语义、source_record_key 隔离、Dify 推送配置

说明：此处“不需要改造”指既有 `unreviewed_pending`、`rectified_suppressed`、`empty_lab_exam` 等跳过语义不回退；本轮仍需要新增 `insufficient_surgery_docs` 跳过原因及其预检/日志展示。

**本规格 v1.1 已修订的关键点**：
- 当前 `PushLog` 没有 `audit_run_mode` 字段；覆盖机制不能只新增 `superseded_by`，必须同步新增并落库 `audit_run_mode`。
- 手动批量推送走 `BulkPushExecutor`，调度走 `PushExecutor`，批量重推走 `PushExecutor.execute_retry()`，单条日志重推当前在 `app/routers/logs.py` 路由内实现；覆盖逻辑和新增跳过逻辑必须覆盖这些入口。
- 规则 2 的真实 builder 在 `app/services/lab_exam_payload_builder.py`，不是只改 `payload_composer.py`；实际 `audit_type_code` 以 `lab_exam_vs_progress_nursing` 为准，不使用文档别名 `jyjc_vs_bcnursing` 作为落库/匹配条件。
- 当前 `scheduler_run_modes.py` 的出院终末 SQL 转换仍识别 `jyjc_vs_bcnursing`；实施时必须同步兼容 `lab_exam_vs_progress_nursing`，否则规则 2 出院终末可能继续按旧日增 SQL 查询。
- `progress_vs_nursing` 在 `discharge_final` 下仍是 `primary` 单源 bundle；`scheduler_run_modes.py` 只是替换 `primary.query_sql` 为 `_PROGRESS_NURSING_DISCHARGE_SQL`，不会产生 `progress` / `nursing` 两个 source。
- `surgery_chain` builder 不能返回 `None, reason`；当前 `compose()` 契约要求返回 `(payload, mr_text)`，跳过应在执行器前置检查或统一跳过策略中完成。
- 新增跳过原因必须同步到预检、日志 label、跳过原因统计和前端展示。
- 覆盖状态必须同步支持日志 API、详情、CSV 导出、删除一致性和筛选。
- `progress_vs_nursing` 存在历史 `audit_type_code` 为空/NULL 的兼容数据；覆盖逻辑、重推逻辑、预检统计不能只按精确 code 匹配。

## 二、改造项 1：出院终末按天关联 Payload 重组

### 2.1 业务需求

**规则 1（progress_vs_nursing）出院终末**：

当前行为：取出住院期间全部病程记录 + 全部护理记录，混在一起发给 AI。

期望行为：按天分组，同一天的病程记录和护理记录关联在一起。AI 能清晰看到"6月11日：这天有哪些病程记录 + 这天有哪些护理记录"，逐天审查。

**示例**：
```
患者 6月11日入院，6月16日出院

当前 Payload（混在一起）：
  [病程] 6/11 首次病程...
  [护理] 6/11 入院护理评估...
  [病程] 6/12 日常病程...
  [护理] 6/12 常规护理...
  [病程] 6/15 术前小结...
  [护理] 6/15 术前护理...

期望 Payload（按天分组）：
  ── 6月11日 ──
  [病程] 首次病程...
  [护理] 入院护理评估...
  ── 6月12日 ──
  [病程] 日常病程...
  [护理] 常规护理...
  ── 6月15日 ──
  [病程] 术前小结...
  [护理] 术前护理...
```

**规则 2（lab_exam_vs_progress_nursing）出院终末**：同理，检验结果、检查结果、病程记录、护理记录按同一天分组关联。

说明：`jyjc_vs_bcnursing` 仅可作为历史文档/别名理解，实施时统一以当前代码中的 `audit_type_code = "lab_exam_vs_progress_nursing"` 为准。

### 2.2 技术实现

**修改文件**：
- `app/services/payload_composer.py` — 修改 `legacy_progress_nursing` builder，并扩展 `compose()` 支持传入 `audit_run_mode`
- `app/services/lab_exam_payload_builder.py` — 修改 `lab_exam_progress_nursing` / `lab_exam_structured_progress_nursing` builder 的出院终末分支
- `app/services/scheduler_run_modes.py` — 规则 2 的出院终末 SQL 转换必须同时兼容 `lab_exam_vs_progress_nursing` 与历史别名 `jyjc_vs_bcnursing`
- `app/services/push_executor.py` — 调用 `compose()` 时传入 `push_config.audit_run_mode`
- `app/services/bulk_push_executor.py` — 调用 `PushExecutor._build_payload_and_mr_text()` 后沿用同一 `audit_run_mode`
- 仅影响 `audit_run_mode == "discharge_final"` 时的 Payload 组装

**实现逻辑**：
```
在 builder 组装 mr_text 时：
  if audit_run_mode == "discharge_final":
      1. 提取每条记录的日期（event_time / 病历标题时间 / 结果时间 / 护理时间）
      2. 按"年-月-日"分组
      3. 每天输出：日期标题 + 该天的各类文书
      4. 按日期升序排列
  else:
      保持当前行为（在院模式不需要按天分组）
```

**compose 签名调整**：
```python
def compose(
    audit_type: AuditTypeConfig,
    bundle: PatientBundle,
    query_date: str,
    audit_run_mode: str = "daily_increment",
) -> tuple[dict[str, Any], str]:
    ...
```

调用点必须从：
```python
payload, mr_text = compose(audit_type, bundle, push_config.query_date)
```

调整为：
```python
payload, mr_text = compose(
    audit_type,
    bundle,
    push_config.query_date,
    audit_run_mode=push_config.audit_run_mode,
)
```

**规则 1 实现要求**：
- `legacy_progress_nursing` 在 `daily_increment` 下保持当前 `build_dify_payload()` / `build_dify_mr_text()` 行为。
- `legacy_progress_nursing` 在 `discharge_final` 下从 bundle 中提取全住院期病程/护理记录，按自然日输出。
- 必须注意：`scheduler_audit_runner.py` 中 `legacy_progress_nursing` 在 `discharge_final` 下虽然走 `load_patient_bundles()`，但 `scheduler_run_modes.py` 只替换 `primary.query_sql` 为 `_PROGRESS_NURSING_DISCHARGE_SQL`，sources 结构仍是 `{"primary": [...]}`，不是 `progress` / `nursing` 两个 source。
- `_PROGRESS_NURSING_DISCHARGE_SQL` 使用 `LEFT JOIN` 将病程与护理按日期合并到同一行；每条 `primary` 记录可能同时包含病程字段（如 `病历文书_完成时间`、`病历标题时间`）和护理字段（如 `护理记录时间`）。按天分组必须基于这些 `primary` 记录的日期字段，而不是读取不存在的 `bundle.sources["progress"]` / `bundle.sources["nursing"]`。
- 如果旧单源路径只有 primary records，也要保持兼容：按记录中的病程/护理时间字段识别并分组。
- 不得输出 `mr_txt` 字段；builder 输出仍统一使用 `mr_text`。

**规则 2 实现要求**：
- 修改位置是 `app/services/lab_exam_payload_builder.py`。
- `scheduler_run_modes.py` 当前只对 `code == "jyjc_vs_bcnursing"` 做 discharge_final SQL 转换；实施时必须改为同时支持 `code in {"lab_exam_vs_progress_nursing", "jyjc_vs_bcnursing"}`，否则当前真实 code 下规则 2 出院终末仍可能沿用日增 SQL。
- `daily_increment` 保持当前同日异常检验/检查上下文选择逻辑，不改变 Dify 当前输入结构。
- `discharge_final` 增加逐日关联结构，建议在结构化 JSON 中新增 `逐日关联` 字段，同时保留现有 `核查信息`、`患者信息`、`检验检查`、`病程`、`护理` 字段，降低 Dify 兼容风险。
- `逐日关联` 的推荐结构：
```json
{
  "逐日关联": [
    {
      "日期": "2026-06-11",
      "检验报告信息": [],
      "检查报告": [],
      "病程记录": [],
      "护理记录": []
    }
  ]
}
```
- `lab_exam_progress_nursing` 文本输入也应按天输出；`lab_exam_structured_progress_nursing` 输出 JSON 字符串。

**关键字段**（用于日期分组的字段）：
- 病程记录：`病历标题时间` / `event_time`
- 护理记录：`护理记录时间` / `EVENT_TIME`
- 检验结果：`结果时间` / `RESULT_DATE_TIME`
- 检查报告：`报告时间` / `REPORT_TIME`

**不影响的部分**：
- 本次不改既有 SQL 查询语义，只改 Payload 组装时的分组逻辑；注意 `scheduler_run_modes.py` 既有逻辑仍会在 `discharge_final` 下替换/注入出院日期 SQL。
- source_record_key 不变
- 跳过策略不变
- 在院模式的 Payload 不变
- Dify 主输入仍必须是字符串，不能把 JSON 对象直传给 Dify
- builder 仍返回 `payload["mr_text"]`，不能返回或写入 `mr_txt`

### 2.3 验收标准

- 出院终末 Payload 中，文书按日期分组，每天有明确的日期标题
- 同一天的病程+护理（规则1）或检验+检查+病程+护理（规则2）关联展示
- AI 能识别按天分组的结构，逐天审查
- 在院模式 Payload 不受影响

## 三、改造项 2：surgery_chain 最小文书种类校验

### 3.1 业务需求

当前行为：surgery_chain 的 `perioperative` 源 kind_filter 接受 `('术后首次病程','手术记录','术前小结')` 三种文书。只要有**任意 1 种**文书存在就会推送。

期望行为：至少需要 **2 种**不同文书类型才推送。只有 1 种文书的跳过。

### 3.2 技术实现

**修改文件**：
- `app/services/push_skip_policy.py` — 新增 surgery_chain 文书种类预检函数
- `app/services/push_executor.py` — 串行/调度路径接入预检
- `app/services/bulk_push_executor.py` — 手动批量路径接入预检
- `app/routers/push.py` — 推送预检接口同步返回该跳过原因
- `app/routers/logs.py` — `_SKIP_REASON_LABELS` 增加中文 label

**实现逻辑**：
```python
def get_surgery_chain_skip_reason(audit_type, bundle) -> tuple[str, str]:
    """返回 surgery_chain 文书不足跳过原因；非 surgery_chain 返回空原因。"""
    payload_cfg = audit_type.payload or {}
    builder = str(payload_cfg.get("builder") or "")
    is_surgery_chain = audit_type.code == "surgery_chain" or builder == "surgery_chain"
    if not is_surgery_chain:
        return "", ""

    records = bundle.sources.get("perioperative")
    if records is None or not records:
        records = (bundle.sources.get("progress") or []) + (bundle.sources.get("surgery") or [])
    records = records or []

    doc_types = set()
    for r in records:
        name = r.get("record_name", r.get("record_type", ""))
        for keyword in ("术前小结", "术前讨论", "手术记录", "术后首次病程", "术后病程"):
            if keyword in str(name):
                doc_types.add(keyword)

    if len(doc_types) < 2:
        return "insufficient_surgery_docs", "围手术期文书不足2种，跳过 Dify 推送"

    return "", ""
```

**接入要求**：
- 不允许 builder 返回 `None, reason`，否则会破坏 `compose()` 契约。
- 新函数放在 `app/services/push_skip_policy.py` 中，建议命名为 `get_surgery_chain_skip_reason(audit_type, bundle) -> tuple[str, str]`。
- `PushExecutor._push_single_record()`：在 `_build_payload_and_mr_text()` 后、Dify 调用前执行 surgery_chain 预检。
- `BulkPushExecutor._process_single()`：在阶段 1 前置检查中执行同样预检，命中后写 skipped PushLog 并 commit。
- `app/routers/push.py` 的 `/api/push/precheck` 也要复用同一判断，避免“预检可推送、实际被跳过”。当前 precheck 路径位于 `push_precheck()`，先调用 `summarize_bundles()`，再叠加 `_load_latest_push_map()`、整改抑制和 `already_succeeded/unreviewed_pending` 统计；surgery_chain 新规则应在该路径中同步更新 `skip_reason_counts/skip_count/pushable_count`。
- 跳过日志必须使用 `create_skipped_push_log()`，保证 `skip_reason`、`audit_run_mode`、`audit_type_code`、`source_record_key` 一致落库。
- 触发条件必须同时兼容 `audit_type.code == "surgery_chain"` 和 `payload.builder == "surgery_chain"`，避免环境配置 code 与 builder 不一致时规则失效。
- 如果 `perioperative` source 存在但为空，仍允许回退检查 `progress + surgery`，保持当前 builder 的容错行为。

**跳过原因**：`skip_reason = "insufficient_surgery_docs"`，label = "围手术期文书不足2种"

**不影响的部分**：
- 出院终末模式同样适用（至少 2 种才推送）
- 其他规则不受影响
- 已有 `empty_lab_exam`、`unreviewed_pending`、`rectified_suppressed` 逻辑不受影响

### 3.3 验收标准

- 只有 1 种手术文书的不再推送，记录 `skip_reason = insufficient_surgery_docs`
- 2 种及以上的正常推送
- 在推送日志中能看到跳过原因

## 四、改造项 3：出院终末覆盖在院质控结果

### 4.1 业务需求

患者 A 在院期间每天被 daily 质控检查（产生多条 PushLog）。患者 A 出院后，discharge 终末质控再检查一次。出院终末结果应标记为"权威结果"，之前的在院结果标记为"已被覆盖"。

### 4.2 数据库变更

**PushLog 新增字段**：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `audit_run_mode` | VARCHAR(32) / String(32) | 运行模式：`daily_increment` / `discharge_final`。默认 `daily_increment` |
| `superseded_by` | INTEGER（可空） | 覆盖此记录的出院终末 PushLog ID。NULL = 未被覆盖 |
| `superseded_at` | DATETIME/TIMESTAMP（可选） | 被覆盖时间。若不实现该字段，规格中的展示仅依赖 `superseded_by` |

**兼容性**：
- SQLite：
  ```sql
  ALTER TABLE push_log ADD COLUMN audit_run_mode VARCHAR(32) DEFAULT 'daily_increment';
  ALTER TABLE push_log ADD COLUMN superseded_by INTEGER;
  ALTER TABLE push_log ADD COLUMN superseded_at DATETIME;
  ```
- Oracle：
  ```sql
  ALTER TABLE MED_PUSH_LOG ADD AUDIT_RUN_MODE VARCHAR2(32) DEFAULT 'daily_increment';
  ALTER TABLE MED_PUSH_LOG ADD SUPERSEDED_BY NUMBER(10);
  ALTER TABLE MED_PUSH_LOG ADD SUPERSEDED_AT TIMESTAMP NULL;
  ```
- `app/models.py` 的 `PushLog` 增加上述字段。
- `app/database.py` 的 `_migrate_push_log_columns()` 增加 SQLite 字段迁移。
- `app/database.py` 的 `_migrate_oracle_alert_columns()` 在 `MED_PUSH_LOG` 迁移项增加 Oracle 字段。
- `app/database.py` 的 `_verify_required_schema()` 增加列存在性检查。
- `app/services/push_log_writer.py` 的 `create_push_log()` 和 `create_skipped_push_log()` 均写入 `audit_run_mode=push_config.audit_run_mode or "daily_increment"`。
- 现有记录：`audit_run_mode` 默认为 `daily_increment`，`superseded_by = NULL` = 未被覆盖 = 当前行为不变。
- 可选索引：`patient_id + visit_number + audit_type_code + audit_run_mode + status`，用于覆盖更新查询。
- 历史出院终末日志如果没有 `audit_run_mode`，默认会被视为 `daily_increment`；本轮自动覆盖只保证新增字段后产生的 `discharge_final` 日志。若必须支持历史出院终末日志重推后触发覆盖，需要从 `source_record_key` 的 `mode::discharge_final::` 前缀或 `request_json` 中额外推断模式。
- `superseded_by` 只允许 `NULL` 或整数 PushLog ID；不得写入空字符串。否则 SQLite 与 Oracle 对空字符串/NULL 的处理差异会导致 `hide_superseded` 筛选行为不一致。

**历史兼容要求**：
- `progress_vs_nursing` 的历史记录可能存在 `audit_type_code = ''` 或 `NULL`。
- 覆盖逻辑若按 `progress_vs_nursing` 匹配，必须复用现有日志/导出查询中的兼容口径：`audit_type_code == 'progress_vs_nursing' OR audit_type_code == '' OR audit_type_code IS NULL`。
- 可选补救方案：上线前执行一次历史回填脚本，将空 code 的 legacy 记录补成 `progress_vs_nursing`；若未回填，则覆盖查询必须兼容空值。
- 历史出院终末日志在新增 `audit_run_mode` 后可能被默认成 `daily_increment`，会被覆盖查询误认为在院记录。覆盖查询必须排除 `source_record_key` 以 `mode::discharge_final::` 开头的记录；更稳妥的方案是上线前执行历史模式回填，将这类日志修正为 `audit_run_mode='discharge_final'`。

### 4.3 覆盖逻辑

**修改文件**：
- `app/services/push_log_supersede.py` — 新增覆盖服务函数，避免逻辑散落
- `app/services/push_executor.py` — 串行/调度路径接入
- `app/services/bulk_push_executor.py` — 手动批量路径接入

**触发时机**：在 `_create_push_log()` 成功写入出院终末 PushLog，`db.flush()` 获得 `log.id` 后执行。应在事务提交前执行，保证出院日志和覆盖标记同事务提交。

**逻辑**：
```python
def mark_daily_logs_superseded(db, discharge_push_log):
    """出院终末推送成功后，标记对应的在院质控结果为已覆盖。"""
    if discharge_push_log.audit_run_mode != "discharge_final":
        return 0  # 只有出院终末才覆盖
    if discharge_push_log.status != "success":
        return 0  # 只有出院终末成功日志才覆盖

    if not str(discharge_push_log.visit_number or "").strip():
        return 0  # visit_number 为空时不做覆盖，避免误匹配

    query = db.query(PushLog).filter(
        PushLog.patient_id == discharge_push_log.patient_id,
        PushLog.visit_number == discharge_push_log.visit_number,
        PushLog.audit_run_mode == "daily_increment",
        PushLog.status == "success",
        PushLog.superseded_by.is_(None),  # 只覆盖未被覆盖的
        ~PushLog.source_record_key.like("mode::discharge_final::%"),
    )

    if discharge_push_log.audit_type_code == "progress_vs_nursing":
        query = query.filter(
            or_(
                PushLog.audit_type_code == "progress_vs_nursing",
                PushLog.audit_type_code == "",
                PushLog.audit_type_code.is_(None),
            )
        )
    else:
        query = query.filter(PushLog.audit_type_code == discharge_push_log.audit_type_code)

    count = query.update(
        {
            "superseded_by": discharge_push_log.id,
            # 如果实现 superseded_at，同步写当前时间
            # "superseded_at": datetime.now(),
        },
        synchronize_session=False,
    )
    return int(count or 0)
```

**接入点要求**：
- `PushExecutor._push_single_record()`：创建成功日志并 `db.flush()` 后调用 `mark_daily_logs_superseded(db, log)`。
- `BulkPushExecutor._process_single()` 阶段 3：创建成功日志并 `db.flush()` 后调用 `mark_daily_logs_superseded(db, log)`。
- `PushExecutor.execute_retry()`：若重推更新的是 `audit_run_mode == 'discharge_final'` 的既有日志，且重推结果为 success，也要在更新该日志后触发覆盖。
- 对新增字段上线前产生的旧日志，`audit_run_mode` 可能缺失或被默认成 `daily_increment`；若需要支持历史出院终末日志重推后触发覆盖，必须从 `source_record_key` 的 `mode::discharge_final::` 前缀或 `request_json` 中推断模式。若不做推断，需在实施说明中明确“仅新增字段后产生的 discharge_final 日志支持重推后自动覆盖”。
- `app/routers/logs.py` 的 `POST /api/logs/{log_id}/retry` 单条重推当前直接在路由内更新既有日志；若保留该实现，也要在该路由中补同样的覆盖逻辑。
- `app/routers/push.py` 的批量重推入口最终调用 `PushExecutor.execute_retry()`，因此覆盖逻辑必须放进执行器，不要只改路由。
- 覆盖只针对 `status == "success"` 的出院终末日志；失败、跳过、取消不覆盖。
- 覆盖更新失败时，建议让当前事务回滚，避免出院日志成功但覆盖标记缺失造成状态不一致。
- Oracle 模式下需实测 `.update(..., synchronize_session=False)` 是否稳定；若出现兼容问题，再切换为更保守的逐条更新或 `synchronize_session="fetch"`。
- 覆盖服务应返回 `superseded_count`；最小实现要求是在执行器日志中记录该 count，不强制本轮新增 `SchedulerHistory` 字段，避免扩大数据库迁移范围。

**匹配条件**：
- `patient_id` 相同
- `visit_number` 相同
- `audit_type_code` 相同（同一规则）
- `audit_run_mode = "daily_increment"`（只覆盖在院结果）
- `status = "success"`（只覆盖成功的记录）
- `superseded_by IS NULL`（只覆盖还没被覆盖的）

**不按 query_date 匹配**：
- 该条属于业务口径，当前建议是：出院终末作为本次住院期间的权威结果，覆盖同患者、同住院次数、同规则下全部 `daily_increment` 成功记录，而不是只覆盖同一天 `query_date`。
- 此口径在实施前需用户最终确认；若业务改为“只覆盖同一天”，则覆盖查询需要补 `query_date` 条件并同步调整 UI 文案。
- 未确认前不得实现覆盖查询条件；第二步覆盖机制实施前必须先定稿此口径。

**不影响的部分**：
- 跳过策略不变
- source_record_key 的 mode 前缀隔离不变
- 在院质控照常执行（只是在院结果会被标记）

### 4.4 API 变更

**`app/routers/logs.py`**：
- `PushLogItem` schema 增加 `audit_run_mode: str`、`superseded_by: int | None`，如果实现 `superseded_at` 也同步增加。
- `/api/logs` 返回结果包含 `audit_run_mode`、`superseded_by`。
- `/api/logs/{log_id}` 详情返回同样字段。
- 增加筛选参数 `?hide_superseded=true` 隐藏已被出院终末覆盖的 daily 记录。
- 可选增加 `audit_run_mode` 筛选参数，用于只看在院或只看出院终末。
- `/api/logs/filters/options` 建议返回 `superseded_count`，供前端显示覆盖统计。

**CSV 导出变更**：
- `app/routers/logs.py` 的 `/api/logs/export/csv` 增加列：`运行模式`、`终末覆盖`、`覆盖者ID`。
- CSV 筛选也支持 `hide_superseded`，并把该筛选条件写入 `record_export_audit()` 的 `filter_criteria`。

**删除一致性**：
- `app/routers/logs.py` 的 `_delete_push_logs()` 删除日志前，应先处理引用关系。
- 如果删除的是出院终末 PushLog，需要将 `PushLog.superseded_by.in_(ids)` 的记录置为 `NULL`，可选同步清空 `superseded_at`。
- 删除 daily 被覆盖记录不需要额外处理。
- 批量删除同时包含 discharge 与其已覆盖 daily 时，执行顺序必须固定为：先清理 `superseded_by` 引用，再删除关联维度/结论/反馈，再删 `PushLog` 本体。

**关联数据策略**：
- 覆盖是“权威结果/展示状态”语义，默认不联动修改既有 `QCFeedback`、`QCFeedbackHistory`、`QCRecordAlertLog`。
- 现有 `rectified_suppressed` 跨模式抑制逻辑保持不变；覆盖不自动清理整改记录。
- 因此已覆盖 daily 上的整改抑制仍可能影响后续推送；该行为需业务确认后再实施。
- 如业务后续要求“终末覆盖后忽略旧 daily 的整改抑制/预警记录”，需单独立项，不纳入本次基础改造。

### 4.5 验收标准

- 出院终末推送成功后，同一患者+规则的在院成功 PushLog 的 `superseded_by` 被设为出院 PushLog ID
- 在院质控仍正常执行（不会因为覆盖标记而停止）
- 查询日志 API 返回 `superseded_by` 字段
- 已覆盖的记录不再触发跳过逻辑变更
- 手动批量推送和调度推送产生的出院终末日志都能覆盖 daily 记录
- 删除出院终末日志后，原 daily 记录的 `superseded_by` 被清空
- CSV 能导出覆盖状态
- 出院终末重推成功后，同样能刷新并覆盖对应 daily 记录

## 九、风险与回滚

| 改造项 | 风险 | 回滚方式 |
|---|---|---|
| 按天 Payload 重组 | AI 可能不适应新的 Payload 格式 | builder 中加开关，可回退为旧格式 |
| surgery_chain 校验 | 误跳过（文书名称匹配不准） | 调整 keyword 列表或降低阈值为 1 |
| 覆盖机制 | 字段 NULL 兼容性 | `superseded_by` 默认 NULL = 无影响 |
| UI 标签 | 前端渲染异常 | 移除标签代码即可 |
| audit_run_mode 落库 | 旧数据无模式值 | 默认 `daily_increment`，历史记录按在院处理 |
| 批量路径遗漏 | 手动推送与调度行为不一致 | 覆盖服务统一函数，PushExecutor/BulkPushExecutor 都调用 |
| 删除出院终末日志 | daily 留下悬挂 `superseded_by` | 删除前清空引用 |
| 历史空 audit_type_code | progress_vs_nursing 覆盖不生效 | 兼容空值查询或先执行历史回填 |
| visit_number 为空 | 误覆盖其他住院次记录 | visit_number 为空时禁止覆盖并记 warning |
| 重推路径遗漏 | 重推成功但覆盖状态不刷新 | 将覆盖逻辑接入 `execute_retry()` 和单条重推路由 |
| query_date 覆盖口径 | 业务理解不一致 | 实施前确认“覆盖全部住院期间 daily”还是“仅覆盖同日” |
| Oracle 批量更新 | 兼容性不稳定 | 在 Oracle 环境专项验证；必要时改逐条更新 |

**回滚注意**：
- 不建议物理删除新增列；回滚代码时可保留 `audit_run_mode`、`superseded_by` 字段，不影响旧逻辑。
- 如果按天 Payload 影响 AI 输出质量，优先通过代码分支关闭 `discharge_final` 按天分组；不要改变 Dify 主输入变量约定。
- 如果 surgery_chain 误跳过，优先调整文书类型关键词，不要绕过统一 skipped 日志写入。

# 前端改进计划

> 文档创建时间：2026-04-06  
> 项目：医疗记录一致性审计系统（Med-Audit）  
> 当前前端规模：index.html 1232行 + app.js 1892行 + app.css 137行

---

## 总体原则

- **渐进式改进**：不引入构建工具，保持当前 Vue 3 全局构建 + Element Plus 技术栈不变
- **功能优先**：先完成高价值功能，再做代码架构重构
- **向后兼容**：每次改动不破坏已有功能
- **医护友好**：界面决策以一线医护/质控人员的使用场景为准

---

## 改进阶段划分

| 阶段 | 主题 | 预估工作量 | 目标完成 |
|------|------|-----------|---------|
| Phase 1 | 高价值 UI/UX 修复（P0） | 小 | 立即执行 |
| Phase 2 | 页面功能增强（P1） | 中 | 近期执行 |
| Phase 3 | 架构拆分重组（P2） | 大 | 中期执行 |
| Phase 4 | 工程化升级（P3） | 超大 | 长期规划 |

## Phase 3 — 代码架构拆分（P2）

> 目标：提升代码可维护性，为长期迭代打好基础

### Task 3.1 引入 ES Module 模块化

**目标**：将 1892 行的 `app.js` 拆分为多个模块文件。

**拆分方案**：
```
static/scripts/
├── app.js               # 主入口，仅保留 Vue 实例创建和全局状态
├── utils/
│   ├── formatters.js    # formatDateTime, severityLabel 等格式化函数
│   ├── api.js           # 所有 axios 调用统一封装
│   └── chart-helpers.js # ECharts 配置生成函数
└── modules/
    ├── auth.js          # 登录/认证/会话恢复
    ├── dashboard.js     # 仪表盘数据与图表渲染
    ├── logs.js          # 推送日志列表/详情/导出
    ├── feedback.js      # 质控反馈
    ├── push.js          # 数据推送/进度轮询
    ├── stats.js         # 数据统计图表
    ├── config.js        # 配置管理（Oracle/Dify/调度等）
    ├── scheduler.js     # 定时任务
    └── admin.js         # 用户/角色/权限/科室管理
```

**改动方式**：
- `index.html` 中 `<script>` 改为 `<script type="module">`
- 各模块导出方法对象，app.js 中 `Object.assign(methods, ...)` 合并

**验收标准**：
- [ ] 所有模块文件正确导入导出
- [ ] 浏览器控制台无模块加载错误
- [ ] 所有功能行为与拆分前一致

### Task 3.3 引入 Day.js 替换手写日期处理

**问题**：`formatDateTime()` 是手写的字符串替换，无法处理时区、相对时间等需求。

- 推送日志列表的时间展示增加相对时间（如 "3小时前"）

**验收标准**：

## Phase 4 — 工程化升级（P3，长期规划）

> 目标：彻底解决无构建工具带来的开发效率问题

### Task 4.1 迁移 Vite + Vue 3 SFC

**技术选型**：
- **构建工具**：Vite 5.x
- **组件化**：Vue 3 SFC（`.vue` 单文件组件）
- **路由**：Vue Router 4
- **状态管理**：Pinia
- **CSS 方案**：保留 Element Plus，新增 CSS Modules

**迁移策略**（渐进式）：
1. 新建 `frontend/` 目录，初始化 Vite 项目
2. 将现有功能逐页面迁移为 `.vue` 组件
3. 后端同时服务 `/` (旧版) 和 `/new/` (新版)
4. 验证新版功能完整后，切换 `/` 指向新版

**验收标准**：
- [ ] 所有页面功能迁移完成
- [ ] 构建产物体积不超过当前 vendor 总大小（约 1.6MB）
- [ ] 首屏加载时间不超过现有版本

## 第三方库引入计划

> 以下库均以不引入构建工具为前提，下载到 `vendor/` 目录本地使用

| 库 | 版本 | 文件 | 用途 | 阶段 |
|----|------|------|------|------|
| **Day.js** | 1.11.x | `dayjs.min.js` | 日期格式化与相对时间 | Phase 3 |
| **Highlight.js** | 11.x | `highlight.min.js` + css | AI 结果 JSON 代码高亮 | Phase 2 |
| **NProgress** | 0.2.x | `nprogress.min.js` + css | 顶部加载进度条 | Phase 2 |

**引入方式说明**：
1. 从官方 CDN 或 npm 下载对应 min 文件
2. 放入 `static/vendor/` 目录
3. 在 `index.html` 中以 `<script src="/vendor/xxx.min.js">` 引入
4. 无需 npm install，保持现有无构建工具架构

## 风险说明

| 风险 | 描述 | 缓解措施 |
|------|------|---------|
| 菜单重组破坏路由 | `switchMenu()` 中的 key 变化影响书签/历史 | 保留旧 key 的别名映射 |
| AI 结果 JSON 格式不稳定 | Dify 返回结构变化导致解析失败 | 增加 try/catch 优雅降级，降级后仍展示原始 JSON |
| ES Module 兼容性 | 极老旧浏览器不支持 | 系统在院内部署，浏览器版本可控，实际风险低 |
| 大规模重构影响测试 | Phase 3/4 改动大，测试成本高 | 每个 Task 单独验收，不一次性合并大 PR |

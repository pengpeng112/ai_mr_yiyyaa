## 0. 复核修订结论（v1.1）

本方案 v1.0 的业务方向正确：系统应拆成两类质控任务：

1. **在院日增量质控**：以当前在院患者为基础，只检查 query_date 当天产生/完成的文书。
2. **出院终末质控**：以 query_date 当天出院患者为基础，检查其住院期间全部历史文书。

但 v1.0 存在若干技术问题，不能直接实施：

| 问题 | 结论 | 修订要求 |
|---|---|---|
| `progress_vs_nursing` daily 改 SQL 不生效 | 原因是 daily 走 legacy `fetch_records(db_cfg)`，不使用 `audit_type.sources.primary.query_sql` | 必须改 legacy 路径的 SQL 来源，或强制 daily 也走 `load_patient_bundles()` |
| EMR 在院方案先拉全量历史文书再 Python 过滤 | 性能风险高 | 必须新增 Vastbase SQL 层按患者+日期查询，不能客户端过滤 |
| `discharge_vs_frontpage` 参与 daily 在院质控 | 逻辑上不合理，在院患者通常无出院记录 | 建议从 daily 调度中排除，只保留 discharge_final |
| `syssvsscbc` daily 只加在院过滤 | 仍不是“昨天文书”，会每天重复扫所有在院手术患者 | 必须明确锚点：昨天手术 或 昨天术后首次病程 |
| `jyjc_vs_bcnursing` 用正则替换 SQL | 风险高，长 SQL 空格/格式不稳定 | 建议使用显式 daily SQL 模板或 source-specific SQL 构造，不用复杂正则 |

### 0.2 修订后的推荐实施路线

建议不要一次性修改 6 条规则，而是分阶段实施：

**第一阶段：先修清晰且低风险的 daily 在院圈人问题**

| 规则 | daily 是否执行 | 修改建议 |
|---|---|---|
| `progress_vs_nursing` | 是 | 修 legacy 路径，只查 `出院日期 IS NULL + 昨天病程` |
| `admission_vs_first_progress` | 是 | 新增 EMR 在院按患者+日期 SQL 查询 |
| `surgery_chain` | 是 | 同上，但需确认文书类型是否足以作为在院日增量 |
| `jyjc_vs_bcnursing` | 是 | 单独实现 daily SQL，不用正则改生产 SQL |

**第二阶段：业务确认后再处理规则适用范围**

| 规则 | 建议 |
|---|---|
| `syssvsscbc` | 暂不直接改。生产基线显示当前在院有主手术患者为 0，需要确认是否按“昨天手术”还是“昨天术后首次病程”作为 daily 锚点。 |
| `discharge_vs_frontpage` | 建议从 daily 调度排除，只在 discharge_final 执行。若必须 daily 执行，需要改成其他在院文书类型，不应使用出院记录。 |

**第三阶段：再收敛 scheduler_daily.audit_type_codes**

推荐配置：

```json

    "surgery_chain"
  ]
}
```

`syssvsscbc` 是否加入 daily，待确认锚点；`discharge_vs_frontpage` 不建议加入 daily。

`scheduler_discharge.audit_type_codes` 可继续保留 6 条。

## 1. 背景与目标

### 1.1 业务需求

系统当前有两类定时质控任务：

| 调度任务 | cron | 模式 | 期望语义 |
|---|---|---|---|
| scheduler_daily | 每天 08:32 | daily_increment | 质控**当前在院患者**昨天新产生的病历文书 |
| scheduler_discharge | 每天 01:22 | discharge_final | 质控**昨天出院患者**住院期间的全部病历文书 |

两类任务都执行相同的 6 条质控规则（audit_type），只是圈定患者的维度不同。

### 1.2 核心问题

当前 `daily_increment` 模式的圈人逻辑不正确——它查的是"昨天有哪些文书"，而不是"昨天在院患者有哪些文书"。导致在院和已出院的患者混在一起质控。

**生产数据实测（query_date=2026-06-13）**：

```
v_qybr 视图 = jhemr.v_zybr（在院患者）UNION ALL jhemr.v_cybr（出院患者）

当前在院患者（出院日期 IS NULL）：     1,150 人
昨天出院患者（出院日期 = 昨天）：         136 人
昨天有新病程记录的在院患者：              185 人
昨天有新病程记录的出院患者：               51 人

当前 daily_increment 实际圈定：         236 人（185 在院 + 51 出院，混在一起）
正确应该圈定：                          185 人（仅在院）
```

### 1.3 目标

```text
daily_increment 模式：
  圈人：v_qybr WHERE 出院日期 IS NULL（当前在院患者）
  取数：这些在院患者在 query_date 当天的文书

discharge_final 模式（已有，不变）：
  圈人：v_qybr WHERE 出院日期 = query_date（昨天出院的患者）
  取数：这些出院患者住院期间的全部文书
```

## 2. 现状分析：6 条规则的当前行为

### 2.1 数据后端分类

| 审计类型 | 后端 | 数据源 | 圈人方式 |
|---|---|---|---|
| progress_vs_nursing | Oracle | v_qybr + v_bcjl + ydhl 护理表 | 文书日期 = query_date |
| jyjc_vs_bcnursing | Oracle | v_qybr + his 检验检查 + v_bcjl + ydhl 护理 | **SQL 本身按出院日期过滤** |
| syssvsscbc | Oracle | v_qybr（UNION ALL 5 台手术）+ v_bcjl | **无日期过滤** |
| admission_vs_first_progress | EMR Vastbase | v_blws（入院记录 + 首次病程） | 文书完成日期 = query_date |
| surgery_chain | EMR Vastbase | v_blws（术前小结/手术记录/术后首次病程） | 同上 |
| discharge_vs_frontpage | EMR Vastbase | v_blws（出院记录 + 首次病程） | 同上 |

### 2.2 逐条问题

#### ① progress_vs_nursing — 缺在院过滤

```sql
-- 当前 daily_increment SQL：
FROM jhemr.v_qybr a
INNER JOIN jhemr.v_bcjl b ON a.患者ID=b.患者ID AND a.次数=b.次数
WHERE {dept_filter}
  AND b.病历标题时间 >= TO_DATE(:query_date, 'YYYY-MM-DD HH24:MI:SS')
  AND b.病历标题时间 < TO_DATE(:query_date, 'YYYY-MM-DD HH24:MI:SS') + 1
-- 问题：没有过滤 a.出院日期 IS NULL，出院患者补写文书也会被查到
```

#### ② jyjc_vs_bcnursing — 方向完全反了（最严重）

```sql
-- 当前 lab 源 SQL（daily_increment 模式实际执行的）：
FROM v_qybr a
INNER JOIN his.LAB_TEST_MASTER lm ON ...
WHERE {dept_filter}
  AND a.出院日期 >= TO_DATE(:query_date, 'yyyy-mm-dd')   ← 查的是昨天出院的患者
  AND a.出院日期 < TO_DATE(:query_date, 'yyyy-mm-dd') + 1
  AND c.ABNORMAL_INDICATOR IS NOT NULL
-- 问题：daily 模式应该查在院患者的检验异常，但实际查的是昨天出院患者的检验异常
-- exam、progress 源同样问题
```

#### ③ syssvsscbc — 无日期过滤

```sql
-- 当前 frontpage 源 SQL：
FROM v_qybr a
WHERE {dept_filter}
  AND a.手术 IS NOT NULL
  AND a.手术日期 IS NOT NULL
-- 问题：没有任何日期条件，会查到所有做过手术的患者（包括已出院的）
```

#### ④⑤⑥ EMR Vastbase 类型 — 缺在院过滤

```
当前 daily_increment：
  Vastbase v_blws WHERE LEFT(文书完成时间, 10) = query_date
  → 查到昨天有文书的全部患者（在院+出院混在一起）
```

## 3. 目标架构

### 3.1 改造后的数据流

```
daily_increment 模式（在院质控）：
  ┌─ Oracle 类型 ────────────────────────────────────────────┐
  │ audit_type_for_run_mode() 注入 a.出院日期 IS NULL        │
  │ → SQL 变为：WHERE 出院日期 IS NULL AND 文书时间=query_date│
  └──────────────────────────────────────────────────────────┘
  ┌─ EMR Vastbase 类型 ──────────────────────────────────────┐
  │ date_dimension="inpatient_date"                          │
  │ → _fetch_inpatient_emr_records()                         │
  │   1. Oracle v_qybr WHERE 出院日期 IS NULL → 在院患者列表 │
  │   2. Vastbase v_blws WHERE patient_id IN (在院患者)      │
  │      AND 文书完成时间 = query_date                        │
  └──────────────────────────────────────────────────────────┘

discharge_final 模式（出院终末质控，不变）：
  ┌─ Oracle 类型 ────────────────────────────────────────────┐
  │ audit_type_for_run_mode() 注入 a.出院日期 = :query_date   │
  └──────────────────────────────────────────────────────────┘
  ┌─ EMR Vastbase 类型 ──────────────────────────────────────┐
  │ date_dimension="discharge_date"                           │
  │ → _fetch_discharged_emr_records()（已有）                 │
  │   1. Oracle v_qybr WHERE 出院日期 = query_date            │
  │   2. Vastbase v_blws WHERE patient_id IN (出院患者)       │
  └──────────────────────────────────────────────────────────┘
```

### 3.2 date_dimension 三值映射

| audit_run_mode | date_dimension | 含义 | 触发场景 |
|---|---|---|---|
| daily_increment | `inpatient_date` | 在院患者 + 当天文书 | 定时调度 scheduler_daily |
| discharge_final | `discharge_date` | 出院患者 + 全部文书 | 定时调度 scheduler_discharge |
| （无/其他） | `query_date` | 按文书日期（原始行为） | 手动推送 / 预检 |

```python
def audit_type_for_run_mode(audit_type, audit_run_mode: str):
    if audit_run_mode == "daily_increment":
```

**新增 `_convert_for_inpatient_mode()` 函数**：

```python
_INPATIENT_FILTER = '\n    AND a.出院日期 IS NULL'

def _convert_for_inpatient_mode(audit_type):
    """daily_increment 模式：注入在院患者过滤条件。"""
    code = getattr(audit_type, "code", "")
    cloned = audit_type.model_copy(deep=True) if hasattr(audit_type, "model_copy") else copy.deepcopy(audit_type)

    if code == "progress_vs_nursing":
        # 策略：在 {dept_filter} 后追加 AND a.出院日期 IS NULL
        primary = (cloned.sources or {}).get("primary")
        if primary and primary.query_sql and "{dept_filter}" in primary.query_sql:
            primary.query_sql = primary.query_sql.replace(
                "{dept_filter}", "{dept_filter}" + _INPATIENT_FILTER
            )
            logger.info("在院模式已注入 progress_vs_nursing 在院过滤")
        return cloned

    if code == "jyjc_vs_bcnursing":
        # 策略：当前 SQL 已按出院日期过滤（方向反了），需要：
        # 1. 移除出院日期条件
        # 2. 追加在院过滤
        # 3. 补充文书/结果日期过滤（因为原来靠出院日期圈人，移除后需要用结果时间圈人）
        _DISCHARGE_DATE_PATTERN = re.compile(
            r'AND\s+a\.出院日期\s*[<>]=?\s*TO_DATE\(:query_date[^)]*\)[^A]*?'
            r'AND\s+a\.出院日期\s*[<>]=?\s*TO_DATE\(:query_date[^)]*\)\s*\+\s*1',
            re.DOTALL
        )
        # 各源的结果日期字段
        RESULT_DATE_FIELDS = {
            "lab": "c.RESULT_DATE_TIME",
            "exam": "nvl(er.REPORT_TIME, em.REPORT_DATE_TIME)",
            "progress": "b.病历标题时间",
        }
        modified = 0
        for source_name, date_field in RESULT_DATE_FIELDS.items():
            source = (cloned.sources or {}).get(source_name)
            if not source or not source.query_sql:
                continue
            sql = source.query_sql
            # 移除出院日期条件
            sql = _DISCHARGE_DATE_PATTERN.sub('', sql)
            # 在 {dept_filter} 后追加在院过滤 + 结果日期过滤
            if "{dept_filter}" in sql:
                result_filter = (
                    f"{_INPATIENT_FILTER}\n"
                    f"    AND {date_field} >= TO_DATE(:query_date, 'yyyy-mm-dd')\n"
                    f"    AND {date_field} < TO_DATE(:query_date, 'yyyy-mm-dd') + 1"
                )
                sql = sql.replace("{dept_filter}", "{dept_filter}" + result_filter)
                source.query_sql = sql
                modified += 1
        if modified:
            logger.info("在院模式已转换 jyjc_vs_bcnursing %d 个源（出院日期→在院+结果日期）", modified)
        return cloned

    if code == "syssvsscbc":
        # 策略：在 {dept_filter} 后追加 AND a.出院日期 IS NULL
        modified = 0
        for source_name in ("frontpage", "first_progress"):
            source = (cloned.sources or {}).get(source_name)
            if source and source.query_sql and "{dept_filter}" in source.query_sql:
                source.query_sql = source.query_sql.replace(
                    "{dept_filter}", "{dept_filter}" + _INPATIENT_FILTER
                )
                modified += 1
        if modified:
            logger.info("在院模式已注入 syssvsscbc %d 个源在院过滤", modified)
        return cloned

    # EMR Vastbase 类型：SQL 为空，不需要 SQL 转换
    # 在院过滤由 data_source_loader._fetch_inpatient_emr_records() 处理
    logger.info(
        "在院模式：audit_type=%s EMR 源由 data_source_loader 跨库查询在院患者",
        code,
    )
    return audit_type
```

**注意事项**：
- `jyjc_vs_bcnursing` 的正则替换需要处理生产 SQL 中不一致的空格（`出院日期>=` vs `出院日期 >=`）
- 正则模式使用 `re.DOTALL` 确保跨行匹配
- 替换后需验证 SQL 语法正确性

### 4.3 改造点 2：data_source_loader.py — 新增在院跨库查询

**新增 `_fetch_inpatient_emr_records()` 函数**：

与已有的 `_fetch_discharged_emr_records()` 结构相同，区别在于 Oracle 查询条件：

```python
def _fetch_inpatient_emr_records(
    root_config: dict,
    source_cfg: dict,
    query_date: str,
    dept_filter: list[str] | None,
    source_name: str,
) -> list[dict[str, Any]]:
    """在院模式：Oracle v_qybr 查在院患者 → Vastbase v_blws 查当天文书。

    与 _fetch_discharged_emr_records 的区别：
    - Oracle WHERE: 出院日期 IS NULL（而非 出院日期 = query_date）
    - Vastbase 查询后需按文书日期过滤（只保留 query_date 当天的文书）
    """
    emr_cfg = ConfigParser.parse_emr_vastbase_config(root_config)
    if not emr_cfg.get("enabled"):
        return []

    oracle_cfg = ConfigParser.parse_oracle_config(root_config)
    if not oracle_cfg.get("host"):
        return []

    dept_list = [str(d).strip() for d in (dept_filter or []) if str(d).strip()]

    # 科室过滤条件（同 _fetch_discharged_emr_records）
    dept_cond = ""
    dept_params: dict[str, Any] = {}
    if dept_list:
        clauses = []
        for i, dept in enumerate(dept_list):
            param_key = f"inpat_dept_{i}"
            clauses.append(
                f'(a."所在科室编码" = :{param_key} OR a."所在科室名称" = :{param_key})'
            )
            dept_params[param_key] = dept
        dept_cond = " AND (" + " OR ".join(clauses) + ")"

    # Oracle 查询在院患者（出院日期 IS NULL = 当前在院）
    sql = normalize_sql(f"""
        SELECT a."患者ID" AS patient_id,
               a."次数" AS visit_number,
               a."住院号" AS admission_no,
               a."患者姓名" AS patient_name,
               a."所在科室名称" AS dept
        FROM jhemr.v_qybr a
        WHERE a."出院日期" IS NULL
          {dept_cond}
    """)

    params: dict[str, Any] = {}
    params.update(dept_params)
    execute_params = build_oracle_execute_params(sql, params)

    conn = None
    cur = None
    patient_keys: list[tuple[str, str]] = []
    patient_info: dict[tuple[str, str], dict[str, str]] = {}
    try:
        conn = get_oracle_connection(oracle_cfg)
        cur = conn.cursor()
        cur.execute(sql, execute_params)
        for row in cur:
            pid = str(row[0] or "").strip()
            vid = str(row[1] or "").strip()
            if not pid:
                continue
            patient_keys.append((pid, vid))
            patient_info[(pid, vid)] = {
                "patient_id": pid,
                "visit_number": vid,
                "admission_no": str(row[2] or ""),
                "patient_name": str(row[3] or ""),
                "dept": str(row[4] or ""),
            }
    finally:
        if cur:
            try:
                cur.close()
            except Exception:
                pass
        if conn:
            conn.close()

    if not patient_keys:
        logger.info("[inpatient_emr] query_date=%s 无在院患者 source=%s", query_date, source_name)
        return []

    # Vastbase 查询文书
    document_kind = str(source_cfg.get("document_kind", "") or "").strip()
    kind_filter = str(source_cfg.get("kind_filter", "") or "").strip()

    docs_by_visit = fetch_emr_documents_by_visits(
        emr_cfg,
        patient_keys,
        document_kind=document_kind,
        kind_filter=kind_filter,
    )

    # 合并患者信息 + 按文书日期过滤（只保留 query_date 当天的文书）
    result: list[dict[str, Any]] = []
    for (pid, vid), docs in docs_by_visit.items():
        info = patient_info.get((pid, vid), {})
        for doc in docs:
            # 在院模式：只保留 query_date 当天的文书
            event_time = str(doc.get("event_time", "") or "")[:10]
            if event_time != query_date:
                continue
            if info.get("patient_name") and not doc.get("patient_name"):
                doc["patient_name"] = info["patient_name"]
            if info.get("dept") and not doc.get("dept"):
                doc["dept"] = info["dept"]
            if info.get("admission_no") and not doc.get("admission_no"):
                doc["admission_no"] = info["admission_no"]
            result.append(doc)

    logger.info(
        "[inpatient_emr] source=%s 在院患者=%d 当天文书=%d query_date=%s",
        source_name, len(patient_keys), len(result), query_date,
```

### 6.1 不受影响的部分

| 历史数据 | 已有 PushLog 不受影响 |

### 6.2 受影响的部分

| 影响项 | 变化 | 风险等级 | 说明 |
|---|---|---|---|
| daily 模式患者数 | 236 → ~185（-21%） | 低 | 减少的 51 人由 discharge 模式覆盖 |
| jyjc daily 语义 | 从出院患者检验→在院患者检验 | **高** | 质控对象完全不同，需确认 |
| syssvsscbc daily 患者数 | 大幅减少 | **中** | 从所有手术患者→在院手术患者 |
| EMR 类型 daily 患者 | 从全部→仅在院 | 低 | 更精确的圈人 |

### 6.3 潜在缝隙分析

```text
患者时间线：
  入院 ──────────────────── 出院
  │                          │
  │←── daily 模式覆盖 ──→│  │←── discharge 模式覆盖 ──→│
  │     （在院期间每天）   │  │  （出院当天全部文书）     │
  │                          │
  ──────────────────────────────────────────────────────────→ 时间
  
缝隙点：出院当天
  - daily 模式：出院日期 IS NULL → 当天已出院，不在院模式范围内
  - discharge 模式：出院日期 = query_date → 当天会覆盖
  
结论：无缝隙。出院当天的质控由 discharge 模式完整覆盖。
```

### 6.4 daily 和 discharge 模式的文书重叠

```text
患者在院期间的文书：
  Day1   Day2   Day3   ...   DayN（出院）
  │      │      │             │
  │ daily│ daily│ daily       │ discharge（查全部N天的文书）
  
重叠：discharge 模式会重新质控 daily 模式已质控过的文书。
这是有意的冗余设计——终末质控是在完整住院背景下的全面审查。
source_record_key 的 mode:: 前缀确保两个模式的推送独立计费，不会互相跳过。
```

| 源 | daily 在院条件 | 日期锚点 |
|---|---|---|
| lab | `a.出院日期 IS NULL` | `c.RESULT_DATE_TIME = query_date` |

至少应先完成以下 SQL 计数验证：

```sql
-- lab daily 目标患者数
SELECT COUNT(DISTINCT a."患者ID" || '::' || a."次数")
FROM jhemr.v_qybr a
JOIN his.LAB_TEST_MASTER lm ON a."患者ID"=lm.PATIENT_ID AND a."次数"=lm.VISIT_ID
JOIN his.LAB_TEST_ITEMS b ON lm.TEST_NO=b.TEST_NO
JOIN his.LAB_RESULT c ON c.TEST_NO=lm.TEST_NO AND c.ITEM_NO=b.ITEM_NO
WHERE a."出院日期" IS NULL
  AND c.RESULT_DATE_TIME >= TO_DATE(:query_date,'yyyy-mm-dd')
  AND c.RESULT_DATE_TIME < TO_DATE(:query_date,'yyyy-mm-dd')+1
  AND c.ABNORMAL_INDICATOR IS NOT NULL
  AND c.ABNORMAL_INDICATOR <> 'N';

-- exam daily 目标患者数
SELECT COUNT(DISTINCT a."患者ID" || '::' || a."次数")
FROM jhemr.v_qybr a
JOIN his.EXAM_MASTER em ON a."患者ID"=em.PATIENT_ID AND a."次数"=em.VISIT_ID
LEFT JOIN his.EXAM_REPORT er ON em.EXAM_NO=er.EXAM_NO
WHERE a."出院日期" IS NULL
  AND NVL(er.REPORT_TIME, em.REPORT_DATE_TIME) >= TO_DATE(:query_date,'yyyy-mm-dd')
  AND NVL(er.REPORT_TIME, em.REPORT_DATE_TIME) < TO_DATE(:query_date,'yyyy-mm-dd')+1;
```

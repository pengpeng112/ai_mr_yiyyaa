## 一、项目目标与范围

### 1.1 项目目标

构建医疗记录一致性审计系统，实现以下主线能力：

1. 从 **PostgreSQL 或 Oracle** 的表/视图查询病历与护理数据。  
2. 按 **查询日期 + 患者** 逐条轮询处理。  
3. 将每位患者的结构化文本推送到 **Dify Workflow**。  
4. 接收大模型比对结果并写回系统数据库。  
5. 通过三种渠道推送审计结果：
   - 系统自带界面
   - 企业微信
   - HTTP 回调接口
6. 在系统界面支持“**确认修改完成**”，并形成可审计留痕。

### 1.2 范围边界

- 本系统负责“抽取、比对、推送、留痕”闭环。
- 不直接修改 HIS 原始业务数据。
- 外部系统联动通过 HTTP 回调完成。

## 二、核心业务闭环

```text
PostgreSQL/Oracle 数据源
        ↓
按日期查询 + 科室过滤
        ↓
按患者逐条轮询
        ↓
组装病程+护理文本
        ↓
推送 Dify Workflow
        ↓
接收 AI 一致性比对结果
        ↓
写入审计库（日志+结果）
        ↓
三路推送（界面/企微/HTTP）
        ↓
人工确认“修改完成”并留痕
```

闭环成功标准：

- 每位患者均有推送记录。
- 不一致结果可在界面查看并确认。
- 三种推送可单独开关、独立失败不互相阻塞。

## 四、数据源接入设计（PostgreSQL / Oracle）

### 4.1 统一抽象接口

```python
class DataSourceClient:
    def connect(self): ...
    def disconnect(self): ...
    def ping(self): ...
    def fetch_patients_by_date(self, query_date: str, dept_list: list): ...
    def fetch_dept_list(self): ...
```

### 4.2 数据源工厂

- 配置 `db_type=postgresql|oracle`
- 由工厂返回对应 client
- 业务层只依赖统一接口

### 4.3 查询要求

必须支持：

- 日期过滤（`query_date`）
- 科室过滤（AI 科室清单）
- 病程与护理关联
- 按患者排序
- 表/视图名可配置

### 4.4 Oracle 已验证 SQL（参考）

```sql
SELECT
  a.患者ID, a.姓名, a.性别, a.年龄, a.科室, a.诊断, a.入院日期,
  b.病历标题时间 AS 病程记录时间, b.记录内容 AS 病程记录内容,
  c.创建时间 AS 护理记录时间, c.护理单类型, c.病情观察及护理措施,
  c.体温, c.心率脉搏, c.呼吸, c.血压, c.血氧饱和度, c.意识神志, c.护士签名
FROM jhemr.v_zybr a
LEFT JOIN jhemr.v_bcjl b ON a.患者ID = b.患者ID AND a.次数 = b.次数
LEFT JOIN ydhl.v_hljl c
  ON c.患者ID = b.患者ID || '_' || b.次数
 AND TO_CHAR(b.病历标题时间, 'yyyy-mm-dd') = TO_CHAR(c.创建时间, 'yyyy-mm-dd')
WHERE a.科室 IN :dept_list
  AND TO_CHAR(b.病历标题时间, 'yyyy-mm-dd') = :query_date
ORDER BY a.患者ID, b.病历标题时间, c.创建时间;
```

## 五、按日期+患者轮询设计

### 5.1 轮询规则

- 一次任务对应一个 `query_date`。
- 先查全量记录，再按 `patient_id` 分组。
- 每位患者合并文本后逐条调用 Dify。
- 每条间隔 `interval_ms`（默认 500ms）。
- 单条失败记录为 failed，不中断后续患者。

### 5.2 触发方式

- 手动触发：用于日常补推、临时执行。
- 定时触发：默认每天固定时刻执行。
- 重试触发：针对失败记录批量重推。

### 5.3 Dry Run

支持仅抽数不推送，用于：

- 验证 SQL
- 预览患者数量
- 调试字段映射

## 七、结果存储与数据模型

### 7.1 push_log（推送日志）

```sql
CREATE TABLE push_log (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  push_time       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  trigger_type    TEXT NOT NULL,
  query_date      TEXT NOT NULL,
  patient_id      TEXT NOT NULL,
  patient_name    TEXT,
  dept            TEXT,
  task_id         TEXT,
  workflow_run_id TEXT,
  status          TEXT NOT NULL,
  error_msg       TEXT,
  elapsed_ms      INTEGER,
  retry_count     INTEGER DEFAULT 0
);
```

### 7.2 audit_result（审计结果）

```sql
CREATE TABLE audit_result (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  patient_id        TEXT NOT NULL,
  patient_name      TEXT,
  query_date        TEXT NOT NULL,
  dept              TEXT,
  workflow_run_id   TEXT,
  is_inconsistent   INTEGER DEFAULT 0,
  severity          TEXT,
  summary           TEXT,
  raw_result        TEXT,
  created_at        DATETIME DEFAULT CURRENT_TIMESTAMP,
  confirmed         INTEGER DEFAULT 0,
  confirmed_by      TEXT,
  confirmed_at      DATETIME,
  confirm_remark    TEXT,
  pushed_wechat     INTEGER DEFAULT 0,
  pushed_http       INTEGER DEFAULT 0,
  pushed_ui         INTEGER DEFAULT 1
);
```

### 7.3 confirm_log（确认留痕）

```sql
CREATE TABLE confirm_log (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  audit_result_id   INTEGER NOT NULL,
  patient_id        TEXT NOT NULL,
  action            TEXT NOT NULL,
  operator          TEXT,
  remark            TEXT,
  operated_at       DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 7.4 notify_log（通知回执）

```sql
CREATE TABLE notify_log (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  audit_result_id   INTEGER,
  patient_id        TEXT,
  channel           TEXT NOT NULL,
  status            TEXT NOT NULL,
  request_body      TEXT,
  response_body     TEXT,
  error_msg         TEXT,
  sent_at           DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 八、三种推送方式设计

### 8.1 自带界面推送

- 结果入库后即在列表展示。
- 默认展示未确认异常记录。
- 支持查看详情与确认操作。

### 8.2 企业微信推送

- 采用企业微信机器人 webhook。
- 触发条件：`is_inconsistent=1`。
- 消息内容包含：患者ID、科室、严重等级、摘要、系统链接。

### 8.3 HTTP 回调推送

- 对接第三方系统（HIS/护理系统等）。
- JSON 标准报文，支持自定义 headers。
- 失败时写 notify_log 并可重试。

### 8.4 严重等级与渠道建议

| severity | 界面 | 企业微信 | HTTP |
|---|---|---|---|
| low | √ | × | × |
| medium | √ | √ | × |
| high | √ | √ | √ |

- 严重等级

- 确认状态
- 确认人、确认时间
- 操作：详情 / 确认完成 / 重新打开 / 重推

### 9.3 “确认修改完成”业务规则

- 仅不一致记录可确认。
- 确认后写入 `audit_result.confirmed=1`。
- 必须记录 `confirmed_by/confirmed_at/confirm_remark`。
- 同时新增 `confirm_log`。
- 支持“重新打开”。

### 9.4 关键接口

```http
POST /api/audit-results/{id}/confirm
POST /api/audit-results/{id}/reopen
```

```json
{
  "operator": "护士长-李某",
  "remark": "已核对并修改完成"
}
```

8. **Dify 调试（debug）**
   - 可视化入参配置
   - 自定义文本输入与样式
   - 原始 JSON 返回查看
   - 结构化解析展示

> 以上能力为“**保留清单**”，迁移后必须逐项可用。

### 17.2 当前前端主要问题（需要优化）

- 单文件过大（模板、样式、逻辑高度耦合）
- methods 过长，接口调用分散
- 状态集中在一个 data 对象，难维护
- 重复逻辑较多（例如调试方法重复定义）
- 菜单切换与数据加载强耦合，测试成本高
- 页面首屏资源较大，不利于后续扩展

### 17.3 优化原则（必须遵守）

1. **保留功能优先**：先等价迁移，不做业务删减。
2. **最小风险拆分**：分阶段拆成模块，不一次性重写。
3. **接口不破坏**：优先兼容现有后端 API。
4. **可回滚**：每个阶段都可退回旧页面。
5. **可验证**：每拆一块，都有对应验收清单。

### 17.6 分步优化计划（避免一次性重构）

#### Step 1：代码解耦（不改 UI）

- 抽离 `api/*`：把 axios 调用从页面逻辑中移出
- 抽离 `utils/*`：时间格式、状态映射、通用提示
- 抽离常量：菜单 key、状态枚举

验收：页面功能不变，接口调用路径清晰。

#### Step 2：页面拆分（保持菜单不变）

- 将 `dashboard/push/logs/stats/...` 各区域拆为独立 View 组件
- 保留原菜单与交互，先完成“结构拆分”等价迁移

验收：每个菜单对应单独文件，功能与原页面一致。

#### Step 3：状态管理下沉

- 引入 Pinia 管理跨页面状态（健康状态、当前数据源、全局时间）
- 页面局部状态仍在组件内，避免过度抽象

验收：状态来源清晰，减少 props/event 透传。

#### Step 4：性能优化

- 路由懒加载
- 图表按需初始化与销毁
- 长表格分页与请求去抖
- 减少重复请求（适度缓存）

验收：首屏更快，切页不卡顿，图表渲染稳定。

#### Step 5：清理与收敛

- 去除重复方法（如重复定义的 debug 方法）
- 统一错误提示与日志上报
- 统一样式变量，减少内联样式

验收：代码规模下降、可维护性明显提升。

### 17.7 前端重构风险与控制

- **风险1：功能回退** → 通过“保留矩阵 + 用例回归”控制
- **风险2：接口不兼容** → 保持旧 API，不先改后端契约
- **风险3：上线中断** → 采用双轨切换：`/` 保留旧版，`/new` 新版灰度
- **风险4：排障困难** → 保留详细日志与页面级错误提示
